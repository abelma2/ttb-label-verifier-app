# Benchmark notes — government-warning checks & the bold question

Why this file exists: the government warning has checkable properties (presence, exact wording,
header ALL-CAPS, "Surgeon General" capitalization, and **two bold rules** — the header must be
bold and the remainder/body may NOT be bold). The text-identity properties are read reliably from
the transcription. **Bold is not** — and this file documents the evidence behind how it is gated.
The production default is now `WARNING_BOLD_POLICY = "header_body_gate"` (`config.py`): a
Pass/Review/Fail gate over BOTH bold rules that auto-passes only when both are confidently
compliant, fails on a confident violation of either, and otherwise routes to a reviewer — see the
"Production posture" section for how the benchmark series led here, and the Conclusion (written
for the prior header-only `confidence_gate`) for why bold is neither ignored nor blindly trusted.

> **Removed 2026-06-11.** The experimental parallel warning-only specialist / crop+variant-A path
> (`USE_PARALLEL_WARNING_CHECK`, `extract_fields_with_warning_check`) and its two benchmark scripts
> (`warning_check_benchmark.py`, `real_backs_flag_benchmark.py`) were deleted from the codebase,
> along with the advisory Cloud Vision object signal (`ENABLE_GOOGLE_VISION`). The sections below
> that discuss them are retained as historical findings, not live features. Production remains
> OpenAI-only with `WARNING_BOLD_POLICY = "header_body_gate"`.

> **Run-artifact paths.** Sections below cite evidence files as `output/<name>` — the path where

## Method

`scripts/benchmarks/model_benchmark.py` runs the app's real extraction prompt across many vision models
and scores the warning observations against ground truth. Two kinds of test case:

- **Controlled (`adversarial/`)** — synthetic labels generated by `scripts/generate_adversarial.py`
  with *font-controlled* headers, so bold is **certain** ground truth:
  - `01_compliant` — ALL-CAPS **bold** header, exact wording
  - `02_titlecase` — title-case header (caps violation)
  - `03_notbold` — ALL-CAPS **regular-weight** header (the key bold test)
  - `04_reworded` — bold header, wrong wording
- **Realistic (`test_labels/baseline_labels/`)** — three compliant front+back bottles
  (spirits, malt, wine). Wording/caps ground truth is known; **bold is NOT certifiable**
  (these are realistic renders, not font-controlled — eyeballing header weight isn't proof).

A separate self-calibrating stroke-width measurement (`scripts/benchmarks/measure_bold.py`) was also
tried and was **inconclusive** — deterministic measurement can't separate bold from regular
at warning-text resolution. That negative result is itself informative.

## What IS reliable

Across every model tested (gpt-4o / -mini, gpt-4.1 / -mini / -nano, gpt-5 / -mini / -nano,
o4-mini, gpt-5.4 / -mini, gemini-2.5-pro / -flash):

- **Wording**: exact transcription, and the reworded fake correctly rejected — `7/7` for all
  frontier models (only the *nano* tier mis-read small print: gpt-5-nano `5/7`, gpt-4.1-nano `6/7`).
- **Caps / title-case**: read reliably from the transcription on the strong models (the mini/nano
  tier occasionally "tidied" a title-case header to caps, missing the violation).
- These ride on *character identity* (a capital `G` is a different glyph from `g`), which vision
  transcription is built for.

## Why bold can't be gated

Bold isn't a character — it's the *rendering* of the same glyphs (heavier strokes). It can't be
transcribed; the model can only opine via `header_bold`. The benchmark shows that opinion is
not reliable on realistic labels:

1. **Controlled images split the models.** Only `gpt-4.1`, `gpt-5`, and `gemini-2.5-pro`
   correctly called `03_notbold` not-bold; gpt-4o, gpt-4o-mini, gemini-2.5-flash, gpt-5-nano,
   o4-mini all said bold=True (would false-pass the violation); the minis abstained.
2. **Realistic labels: the best models disagree, even after an improved prompt.** On the three
   compliant baselines (with an explicit "compare header vs body stroke weight" prompt):

   | label | gpt-4o | gpt-4.1 | gpt-5 | gemini-2.5-pro |
   |-------|--------|---------|-------|----------------|
   | baseline_1 | bold | bold | bold | bold |
   | baseline_2 | not  | not  | not  | bold |
   | baseline_3 | not  | not  | bold | bold |

   Consensus only on the clearly-bold label; the subtle ones are contested.
3. **`trust_model` false-fails compliant labels.** With bold gating ON, `gpt-4.1` (the best,
   fastest reader) read all three real baselines as not-bold and **FAILED all three compliant
   bottles** — every other field correct.
4. **Self-reported confidence is not calibrated.** Asked for a `header_bold_basis`, models emit
   a confident justification that mirrors their boolean — e.g. gpt-4o returns
   `bold=True [high]` "header strokes visibly thicker than body text" on `03_notbold`, the
   *known regular-weight* header. So a confidence gate would launder confident errors into passes.

## Conclusion — why bold is *confidence-gated*

Bold is a **mandatory** warning requirement, but the benchmark shows model bold judgment is
**not reliable enough to trust outright** on realistic labels: models disagree with one another
and report high confidence even when wrong. The app therefore neither ignores bold nor blindly
trusts it — it **confidence-gates** the model's bold judgment and **fails closed** when bold
cannot be confirmed. The app does *not* ignore bold; it checks bold and fails when the required
formatting cannot be confirmed.

- `WARNING_BOLD_POLICY = "confidence_gate"` (the **prior** submitted-demo default — **since
  superseded by `header_body_gate`**, which requires **HIGH** confidence on both bold rules to
  PASS; see "Production posture" below. The medium/high-passes behavior described next is this
  prior header-only mode, **not** the current production gate) — `header_bold` True
  with medium/high confidence passes; False with medium/high confidence fails; a null read or
  low confidence fails with *"could not verify ... submit a clearer label image"* (an automated
  inability-to-verify, **not** human review).
- Trade-off (documented, not hidden): because confidence is imperfectly calibrated, a model that
  confidently mis-reads a compliant header as not-bold will false-fail that label. The
  alternative — trusting the read — would false-PASS real violations. Fail-closed is the
  defensible choice for a compliance screen.
- `WARNING_BOLD_POLICY = "note"` is kept for benchmarking/debugging (bold recorded as telemetry,
  not gated). Revisit the gate (e.g. its thresholds) only if a future benchmark shows reliable
  bold discrimination **on realistic (not just synthetic) labels**, with font-controlled
  realistic fixtures and multi-sample stability.

## Model / latency takeaways & the 5× stability pass

A broad single-run comparison plus a **5× stability pass** (`scripts/benchmarks/stability_benchmark.py`)
settled the default extraction model:

- **`gpt-5.4-mini` is the default.** Over 5 runs/case it caught the not-bold adversarial
  (`03_notbold`) **5/5**, passed compliant `01` **5/5**, failed title-case/reworded **5/5**, and
  passed the realistic baselines **14/15** — with stable bold reads (`03_notbold` → False×5;
  baselines → True×5). ~4.2s/label.
- **`gpt-4.1`** (the prior default) catches `03_notbold` but **false-fails every realistic
  baseline (0/15)** — it reads their bold headers as not-bold. Accurate on stark synthetic
  contrast, over-strict on real labels.
- **`gpt-5.5`** is the accuracy ceiling — same behavior as gpt-5.4-mini, ~40% slower; reference,
  not default.
- **`gpt-4o` / `gpt-4o-mini` / `gpt-4.1-mini` / `o4-mini`** are bold-blind — they report
  `bold=True` on everything, so they **false-pass `03_notbold`**. **`gpt-5.4-nano`** is erratic
  (abstains on bold, false-fails the compliant case). None are usable as the default.
- **nano** models (gpt-4.1-nano, gpt-5-nano) also mis-read core fields (brand, class, wording).
- **gemini-2.5-pro** matches accuracy but is ~3–4× slower (~10s) and intentionally not the default.

Note: `gpt-5.4`/`gpt-5.5` reject `reasoning_effort="minimal"` (they need `"low"`); `extraction.py`
handles this in `_model_params` and defensively in `_create_with_fallbacks`.

## Experimental: parallel warning-only specialist

`USE_PARALLEL_WARNING_CHECK` (off by default) runs a narrow warning-only call in parallel with
the full extraction and lets it own `government_warning`. The first run
(`scripts/benchmarks/warning_check_benchmark.py`) preserved adversarial accuracy (4/4, caught `03_notbold`)
and added ~0 wall-clock (it hides behind the slower full call), but gave no measurable gain over
the single-model read and doubles per-label cost — so it stays off pending a stability/second-model
evaluation.

## Side experiments: can another vision approach do bold (or beat the extractor)?

A series of **benchmark-only** experiments tested whether a different Google/OpenAI vision
approach could replace the extractor or solve bold. **None is wired into the app or the verdict**;
all keep `extraction.py` / `verification.py` / `app.py` and `WARNING_BOLD_POLICY` untouched.

- **A — Cloud Vision LABEL_DETECTION / OBJECT_LOCALIZATION** (`google_vision.py`): fast (~0.2s)
  object/container/barcode context (bottle vs can, wine/beer/spirits, 1D/2D barcodes), accurate as
  a *scene* signal but with **no regulatory content** (never sees wording/caps/bold/ABV). Advisory
  only, behind `ENABLE_GOOGLE_VISION` (off by default).
- **B — Cloud Vision DOCUMENT_TEXT_DETECTION (OCR)**: high-quality, fast (~0.25s) raw text that
  preserves case and captures the warning wording verbatim even on real photos — but **cannot see
  bold** (`03_notbold` OCR text is byte-identical to `01_compliant`). Useful only with a parser.
- **B2 — OCR → text-only parser → canonical schema** (`scripts/benchmarks/ocr_parse.py`): ties the vision extractor on
  flat fields (beverage type, ABV, caps, net contents) but is **worse on layout-dependent fields**
  (name/address, class) and on warning wording on real photos (OCR character errors → false
  wording failures); no speed/cost win; bold unknown by design. **Not adopted.**
- **C — Gemini structured extraction** (`scripts/benchmarks/gemini_extraction.py`): gemini-2.5-flash agrees with
  gpt-5.4-mini on the substantive readable fields, but is slower (~9.5s vs ~5.6s), had transient
  availability errors, and **disagrees on bold ~half the time**. gpt-5.4-mini stays the default.

### B3 — OCR-locate → crop → bold specialist (recorded result)

Pipeline (`scripts/benchmarks/bold_specialist.py`, `scripts/benchmarks/bold_specialist_compare.py`): Cloud Vision locates the
warning paragraph → tight, upscaled crop → a gpt-5.4-mini call answering only *"are the header
strokes visibly thicker than the body?"* (the deep-research "more pixels on the warning" lever).

- **Upside:** OCR localized the warning 16/16 on real back labels; the high-res crop **improved
  visibility and reduced some false-FAILS on compliant labels** (it fixed full-image false-fails on
  `test_1`/`test_2`), while still calling the stark `03_notbold` not-bold.
- **Controlled negative testing is the decisive result.** A font-controlled set of warning blocks
  (`scripts/benchmarks/generate_bold_safety.py` → `bold_safety/`: bold-compliant / regular-weight not-bold /
  title-case / all-bold-body, each × clean·lowres·jpeg·rotate+blur·curved) measured the dangerous
  direction: **60% FALSE-PASS on not-bold / all-bold violations** (6/10 definite negatives) — including
  a confident, "clear" false-pass on a **clean** regular-weight header. False-fail on genuinely-bold
  headers was 20%; the model **never abstained** (0 null) and several false-passes were "clear" at
  medium/high confidence, so legibility gating would not catch them.
- **Therefore B3 is UNSAFE as an automated compliance gate** — it would pass real violations. The
  crop improves *perception* but not *honesty* (the model reports its prior "warning headers are
  bold" rather than the actual strokes). **B3 is NOT integrated into the app or verifier.**

### Bold prompt experiment (prompt-only, `scripts/benchmarks/bold_prompt_experiment.py`)

Tested whether a stricter **forensic** prompt makes gpt-5.4-mini's bold read less willing to say
`true` without proving the header is thicker than the body. Four prompts on the controlled set
(adversarial + `bold_safety/` + baseline backs), strict JSON, **no cropping**: `baseline` (≈ the
production prompt, still lists "darker") · `forensic` (drops "darker", forces header-vs-body,
anti-bias) · `forensic+neg` (crisp negatives) · `forensic+neg+choice` (model classifies an A–E
visual relationship; code maps A→true, B/C/D→false, E→null, so it never picks the conclusion).

- **Robust finding:** the production-style `baseline` false-passes heavily (~55–82% across runs);
  **every forensic variant cuts false-pass dramatically (~0–18%)** — chiefly by forcing the
  header-vs-body comparison and removing the "darker" shortcut. (Baseline's 80–100% false-pass on
  the **all-bold-body** case is the tell: it answers "looks bold" without comparing to the body.)
- **Not robust:** *which* forensic variant is best. **Run-to-run variance is large** (bold judgment
  is nondeterministic) — across two runs the same variant's false-pass swung 10–30 points, as big as
  the gaps between variants. The multiple-choice variant did **not** clearly beat plain `forensic+neg`.
- **No variant is gate-safe.** None reliably reaches ~0 false-pass; residual false-passes concentrate
  on **low-resolution** images, where the model commits to "A" (thicker) at *medium* confidence
  instead of "E" (can't compare). Per the safety criterion, a non-trivial false-pass →
  **bold cannot be auto-passed.**

`forensic+neg` / `forensic+neg+choice` is a strictly better **observation** prompt than the current
wording (worth adopting if the bold-observation prompt is ever revised — it makes the fail-closed
posture safer), but it does not make bold auto-passable. Ranking the forensic variants reliably
would need a multi-run stability pass; the gate decision is already settled.

**Recorded conclusion:**
- Forensic prompts **materially improve the bold observation** (false-pass ~55–82% → ~0–18%, by
  forcing the header-vs-body comparison and dropping the "darker" shortcut).
- The **baseline production-style wording is too permissive on subtle negatives** — it false-passes
  regular-weight and all-bold-body headers because "looks bold / darker" is enough for it.
- **Run-to-run variance is large** — the bold judgment is nondeterministic, so a single run cannot
  rank the forensic variants.
- **Even the best prompt still has non-zero false-pass risk** (residual false-passes on low-res,
  where the model says "thicker" at medium confidence instead of abstaining).
- **Therefore bold remains fail-closed / reviewer-confirmed and is NOT auto-passed.** No prompt
  variant is wired into production; `WARNING_BOLD_POLICY` is unchanged.
- A **5× stability pass is deferred** — it would refine *which* forensic prompt is best (a rigorous
  appendix only) and would **not change the production decision**.

### Production confidence_gate measured on the controlled set (diagnostic)

Ran the **actual production path unchanged** — `extract_fields` (production prompt + gpt-5.4-mini)
→ `verify_label_only` (live `WARNING_BOLD_POLICY="confidence_gate"`) — on `bold_safety/`, 3 runs/
image, recording the government-warning field verdict (`scripts/benchmarks/confidence_gate_safety.py`,
`artifacts/confidence_gate_safety_results.*`).

| measured on bold_safety (3×) | result |
|---|---|
| **FALSE-PASS — all-bold-body violations** (warning verdict = PASS) | **14/15 (93%)** |
| **FALSE-PASS — not-bold headers** (verdict = PASS) | **5/15 (33%)** |
| FALSE-FAIL — compliant bold (verdict = FAIL) | 3/15 (20%) |
| title-case (caps violation) | FAIL 15/15 (deterministic caps check works) |

Warning verdicts over all 60 runs: **PASS 31 · REVIEW 1 · FAIL 28**. Eight distinct violations
auto-passed ≥1× (all 5 bold-body + not-bold clean/lowres/rotblur).

**Yes — the current production path auto-passes bold violations**, two ways:
1. **All-bold-body (~93%) is a STRUCTURAL gap:** `_check_warning` checks the *header's* `header_bold`
   only — it does **not** implement §16.22's "the remainder may not be bold." An all-bold warning
   (header bold AND body bold) reads `header_bold=True` → confidence_gate PASS, even though it is a
   violation. The verifier cannot catch the remainder-bold rule today.
2. **Not-bold headers (~33%):** the model false-reads a regular-weight header as `header_bold=True`
   on ~1/3 of runs, and confidence_gate trusts a confident-True → PASS.

Diagnostic only — nothing was changed. It **confirms and sharpens** the conclusion: the implemented
confidence_gate *does* trust bold as an automated pass (on confident-True), so it does **not** fully
meet the stated "never auto-pass bold on model judgment" posture. Closing that gap is a future
decision (e.g. a never-auto-pass policy such as `"review"`, and/or adding a remainder-not-bold check).
`WARNING_BOLD_POLICY` is left unchanged here; the deterministic wording/caps checks remain reliable.

### Reference-image warning comparison (diagnostic)

Tested whether comparing a label's warning to a known-good **reference image**
(`adversarial/Correct_Goverment_Warning.png`) beats deterministic text comparison
(`scripts/benchmarks/reference_warning_compare.py`, `artifacts/reference_warning_compare_results.*`). Three
approaches on adversarial + baseline backs (7 targets, single run): **A** reference-image vs target
(gpt-5.4-mini, strict JSON) · **B** model transcribes → Python compares to `config.GOVERNMENT_WARNING`
+ caps check · **C** Cloud Vision OCR → same Python compare. B/C do not assess bold.

| dimension | A reference-image | B transcribe+Py | C OCR+Py |
|---|---|---|---|
| wording | 5/7 (71%) | **7/7 (100%)** | **7/7 (100%)** |
| caps | 7/7 | 7/7 | 7/7 |
| bold | 7/7 (one run only) | n/a | n/a |
| overall | 6/7 | 6/7 | 6/7 |
| latency | 3.55s | 2.31s | **0.24s** (C OCR) |

- **The reference image did NOT help wording — it hurt it.** A conflated wording with formatting: it
  false-FAILED wording on the title-case image and a compliant baseline (counting the visual caps/
  layout difference as a wording mismatch). Deterministic text compare (B/C) is **exact** on wording
  (100%) and cleanly separates wording from caps.
- **Caps:** all three correct, but B/C get it deterministically from the transcription/OCR (safer).
- **Bold:** A scored 7/7 in this single run, **but do not trust that** — every prior bold experiment
  showed large run-to-run variance and 33–93% false-pass, so one clean run is run-luck, not a safe
  capability. B/C (correctly) do not attempt bold from text; their only "false-pass" is `03_notbold`
  on the *overall* roll-up — the expected bold-blindness, which is exactly why bold is a separate
  fail-closed check.
- **Safer than deterministic text compare? No.** Asking the model to "decide match" is strictly less
  safe and less accurate on wording/caps than a Python string/caps comparison against the canonical
  text. The reference image is at best a **benchmark fixture / prompt aid**, not a production gate.

**Conclusion:** keep the production source of truth as the canonical warning **string** in `config.py`
+ deterministic Python wording/caps comparison; bold stays fail-closed / reviewer-confirmed. The
reference image stays **benchmark-only**.

### B4 — crop + forensic+negative prompt + body check (diagnostic)

B4 = B3's OCR-locate/crop/upscale + the forensic+negative stroke-weight prompt, with an extended
schema that also reports `body_appears_bold` so §16.22's "the remainder may not be bold" can be
checked. Run on bold_safety + adversarial + baseline/real backs (40 targets, single run).
`scripts/benchmarks/bold_b4_crop_forensic.py`, `artifacts/bold_b4_crop_forensic_results.*`.

| header_bold | B3 crop+old | B4 crop+forensic |
|---|---|---|
| false-pass — not-bold | 3/6 (50%) | **0/5 (0%)** |
| false-pass — all-bold-body | 5/5 (100%) | 1/5 (20%) |
| false-fail — compliant | 2/28 (7%) | **12/28 (43%)** |

- **The forensic prompt on the crop slashed false-passes** (not-bold 50→0%, all-bold-body 100→20%) —
  far safer than B3's old prompt, and it even abstained (null) on the stark `03_notbold`.
- **The new `body_appears_bold` check closes the all-bold gap.** The combined **§16.22-aware gate**
  (pass only if header clearly thicker AND body not bold AND verifiable) had **0/11 false-pass on the
  controlled violations** — the first approach to catch the all-bold-body violation (via the
  independent body check; body_appears_bold accuracy ~89%) that confidence_gate (93% false-pass) and
  every prior method missed.
- **But B4 is over-strict: ~43% false-fail on compliant bold** (especially real photographed backs,
  which it now reads as not-bold). It is the inverse of B3 (permissive / high false-pass) — safe but
  impractical.

**Verdict:** B4's combined gate is the **safest bold approach yet (0% false-pass this run)**, but
(a) it is a **single run** and bold judgments have large documented variance, so 0/11 is **not** proof
of near-zero risk; (b) the **43% false-fail** makes it unusable as an auto-pass gate (it would reject
~half of compliant labels). Per the safety rule, **B4 stays benchmark-only — not an automated gate.**
Its genuinely useful output is the `body_appears_bold` flag as **reviewer telemetry** for the
all-bold-body violation the production verifier does not check today.

### Stricter bold gate: current (medium+high) vs high-only (re-scored, no new API calls)

Re-scored the bold gate from `artifacts/confidence_gate_safety_results.json` (production reads on the
controlled `bold_safety` set, 3 runs/image) under two rules — **current** (`header_bold=True` +
conf∈{medium,high} → auto-pass) vs **high-only** (+ conf==high). `scripts/benchmarks/bold_high_only_gate.py`,
`artifacts/bold_high_only_gate_results.*`.

| false on bold_safety | current | high-only |
|---|---|---|
| false-pass — not-bold headers | 6/15 (40%) | **0/15 (0%)** |
| false-pass — all-bold-body | 14/15 (93%) | 5/15 (33%) |
| false-fail — compliant bold | 3/15 (20%) | 9/15 (60%) |

- **High-only eliminates not-bold false-passes** (40%→0%): all 6 current ones were *medium*-confidence
  (the model never high-confidently mis-read a regular header as bold).
- **But it does NOT close the all-bold-body gap.** 5/15 boldbody runs still false-pass at **HIGH**
  confidence — the header genuinely *is* bold, so `header_bold=True/high` is "correct" for the header;
  the violation is the bold *body*, which a header-only gate never checks. Of the 20 current
  false-passes, 15 were medium (fixed) and **5 were high — all all-bold-body** (not fixed).
- **And it triples false-fails on compliant bold** (20%→60%): 6/15 compliant labels that pass at
  medium confidence would newly fail/review (worse on real photos, which read at lower confidence).

**Verdict:** high-only is a real but partial safety gain — it removes medium-confidence false-passes,
but leaves the **structural all-bold-body false-pass (33%, all high-confidence)** and roughly triples
false-fails. It is **not** enough to auto-pass bold. The all-bold-body gap needs the verifier to check
that the *remainder* is not bold (cf. B4's `body_appears_bold`); confidence-tightening alone can't.
Bold stays fail-closed / reviewer-confirmed.

### Production posture — adopted the structural body-check (`header_body_gate`, new default)

The benchmark series above converged on one structural finding: the old header-only
`confidence_gate` never inspected the body weight, so it **auto-passed ~93% of all-bold-body
violations** (the "FALSE-PASS — all-bold-body" row, and the residual high-confidence false-passes
that high-only re-scoring could not fix). B4 showed the fix is a verifier-side body check
(`body_appears_bold`: 0% gate false-pass), at the cost of more false-fails. The production
default now implements exactly that, as `WARNING_BOLD_POLICY = "header_body_gate"`:

- **Auto-check warning wording and capitalization** deterministically (they ride on character
  identity — reliable across models). Unchanged.
- **Bold is now Pass/Review/Fail on BOTH visual rules of 27 CFR 16.22** — the header must be
  bold AND the remainder/body may NOT be bold — judged from the extractor's
  `header_bold`/`header_bold_confidence` **and** the new `body_bold`/`body_bold_confidence`.
  The warning **PASSES only when** `header_bold=True` at high confidence AND `body_bold=False`
  at high confidence; it **FAILS** on a high-confidence violation of either; everything else
  (null / medium / low on either field) **routes to needs-review**. `header_bold=True` alone no
  longer passes — closing the structural all-bold-body gap the header-only gate left open.
- **Still do not auto-pass bold on a low-confidence or single-field judgment.** B3's 60%
  false-pass and the all-bold-body data remain the evidence; the gate only auto-passes when both
  fields are confidently compliant, and otherwise fails closed or routes to a reviewer.
- `"confidence_gate"` (the prior default, header-only) and the other modes are retained for
  benchmarking and A/B comparison via the `WARNING_BOLD_POLICY` env override.

### `medium_pass_gate` — does relaxing the PASS gate to MEDIUM confidence help? (tested, rejected as default)

`header_body_gate` (default) auto-PASSES only when **both** bold rules are satisfied at **HIGH**
confidence; a medium-confidence-but-compliant read goes to REVIEW. That conservatism produces
*false reviews* on clean labels (the bold read is often medium-confidence even when the label is
fine). `medium_pass_gate` (`scripts/benchmarks/medium_pass_gate_compare.py`,
`output/medium_pass_gate_compare_*`) tests whether accepting **medium-or-high** confidence on the
PASS gate is a net win. It is **identical to `header_body_gate` except** the PASS gate also accepts
medium confidence; **FAIL is unchanged** (still a high-confidence violation only), so by construction
it can never soften a high-confidence violation that `header_body_gate` fails.

Method (clean A/B): each image is extracted **once** with the production schema, then **both**
policies are applied to the *identical* `government_warning` observation — so any verdict difference
is purely the policy, with zero extraction noise between the columns. 3 rounds over the 3 clean
baselines + `adversarial/` (01–04) + the `bold_safety/` controlled set (all-bold-body / compliant /
not-bold / title-case, clean+lowres). (2 of 45 rounds errored on a transient API 500 from concurrent
runs; both in clear-signal classes — findings unaffected. The FAIL-parity is also proven
deterministically in the unit tests.)

| | header_body_gate (default) | medium_pass_gate |
|---|---|---|
| **clean baselines** (8 valid reads) | PASS 4 · REVIEW 4 · FAIL 0 | **PASS 8 · REVIEW 0 · FAIL 0** |
| not-bold violation (`notbold`, P/R/F) | 0 / 3 / 6 | **2** / 1 / 6 |
| all-bold-body violation (P/R/F) | 2 / 2 / 2 | 2 / 2 / 2 (identical) |
| title-case / reworded | FAIL (deterministic) | FAIL (deterministic) |
| **false-passes on should-FAIL violations** | **2** (both all-bold-body, HIGH-confidence misreads — irreducible) | **4** (the same 2 **+ 2 new** `notbold_lowres`, MEDIUM-confidence misreads) |

**Findings:**
- **It reduced false reviews on clean labels** — all 4 baseline REVIEWs (each a medium-confidence,
  both-rules-satisfied compliant read) became PASS; clean-baseline reviews 4→0, zero new FAILs. (The
  separate `run_eval` completeness pass agreed directionally: clean baselines 2/3 → 3/3 fully PASS.)
- **But it introduced new false PASSES on not-bold warning violations** — on a low-res `notbold`
  image the model mis-read the regular-weight header as `header_bold=True[medium]` with
  `body_bold=False[medium]`; `header_body_gate` routes that medium read to a human (REVIEW), while
  `medium_pass_gate` **auto-passes a real violation** (false-passes 2→4). This is the exact
  medium-confidence-misread surface the high-only gate was designed to keep closed (cf. the
  "Stricter bold gate" re-scoring above: all not-bold false-passes were medium-confidence).
- **Therefore it is NOT recommended as the default.** It trades reviews-of-compliant-labels for
  passes-of-violations — the wrong direction for a fail-closed compliance gate (a false review costs
  a human a second look; a false pass ships a non-compliant warning). The better way to cut the false
  reviews is to raise *read* confidence (the crop/specialist legibility work above), not to trust
  medium-confidence reads.
- **`header_body_gate` remains the default because the warning bold judgment must stay conservative.**
  Production auto-PASS still requires HIGH confidence on **both** bold rules; medium confidence is
  **never** sufficient for a production PASS.

**Status:** `medium_pass_gate` is kept as an **experimental, env-selectable option only**
(`WARNING_BOLD_POLICY=medium_pass_gate`), with full truth-table unit tests (incl. a FAIL-parity test
vs `header_body_gate`) in `tests/test_verification.py`. It is **not** wired in as the default and
production behavior is unchanged. Reasonable to opt into only in a low-stakes context with a
guaranteed downstream human pass (e.g. a batch pre-screen) — never as the authoritative gate.

(~5s, no retry) failing safe to the full-image observation/review — the natural integration point is
the existing `USE_PARALLEL_WARNING_CHECK` path (`extract_fields_with_warning_check`), validated via
`warning_check_benchmark.py` first.

**Status (adopted 2026-06-09).** Two production changes followed from this result:
1. **Variant-A bold wording is now the production wording** in BOTH `extraction.py::_PROMPT` and
   `_WARNING_CHECK_PROMPT` (kept in sync per the inline note): the "darker" shortcut — the measured
   cause of the all-bold-body false-passes — was removed; the de-prime clause ("Do NOT infer bold
   from capitalization, font size, darkness, contrast, or expectation") and abstention license
   ("If uncertain, use null and low confidence") were added, and the body_bold instruction no longer
   primes the expected answer with the federal rule. Regression-guarded by
   `test_bold_prompts_are_deprimed_variant_a_style`.
2. **The crop+A pipeline is wired into `extract_fields_with_warning_check`** behind
   `USE_PARALLEL_WARNING_CHECK` (still **off by default**): OCR-locate (back image first) -> 8%-pad
   crop upscaled to 1000px -> warning-only call on the crop, in parallel with the main extraction.
   Strict fallback rule: the crop result is used ONLY if it finished before the main extraction was
   ready (0.1s scheduling grace) AND is usable (warning present with text) AND passes the
   **safety-ordered merge rule** (`_choose_warning_evidence`). The rule evolved across two
   validation rounds: v1 (crop replaces wholesale) let a medium-confidence crop soften a
   high-confidence full-read FAIL on `03_notbold` to REVIEW; v2 (never touch a FAIL) closed that but
   locked in a confidently-WRONG main false-FAIL on the compliant CONTROL-CASE that the crop had
   read correctly as PASS. The current rule applies the project's witness-disagreement principle:
   both observations are ranked by the verdict the production `_check_warning` would give —
   deterministic text FAILs (wording/caps) are untouchable; a main **bold-only FAIL** contradicted by
   **fully high-confidence compliant** crop evidence merges to a **witness-disagreement
   needs_review** (`bold_witness_disagreement`, settable only by deterministic code — `_coerce`
   strips it from model output — and judged by `_check_warning`: "warning bold evidence disagrees
   between the full-label and crop reads"); medium/low/null crop evidence never challenges a FAIL;
   the crop may escalate (REVIEW->FAIL, PASS->FAIL) and demotes a main PASS to review only on a
   concrete contradictory/uncertain observation; REVIEW->PASS only with fully high-confidence
   compliant bold evidence and no high-confidence contradiction. So the crop never extends
   wall-clock, never auto-passes a contradiction, and a high-confidence contradiction always reaches
   a human; otherwise the main extraction's own government_warning is the evidence, with NO extra
   fallback call. Cloud Vision/pillow unavailable -> soft-disables to the main path.
   `WARNING_BOLD_POLICY` is unchanged. Run `warning_check_benchmark.py` on this wiring before
   enabling the flag.

   **Post-review hardening (2026-06-09, xhigh code review — 15 findings, all fixed).** The merge
   rule was hardened against the review's three confirmed sev-4 gaps, all in TEXT handling: (1) a
   crop whose OCR bbox TRUNCATED the warning could escalate a wording FAIL/REVIEW over a perfect
   full-label read — now crop text-cause problems (wording/caps) never outrank the main read; (2) a
   main ABSENCE FAIL was untouchable even when the crop had located+transcribed the warning — now a
   presence contradiction merges to a witness-disagreement review; (3) the REVIEW->PASS promotion
   ignored WHY the main read reviewed, so a wording-near-miss review could auto-pass on memorized
   canonical crop text — now only bold-cause reviews can be promoted. Root-cause fix for the
   reason-string fragility: `FieldResult.cause` (machine-readable: absence/wording/caps/bold/
   disagreement/low_confidence) replaces ALL reason sniffing in the merge layer (also fixes the
   escalated-PASS reason falsely counting as concrete doubt, and confidence_gate's fail-closed FAIL
   falsely triggering disagreement). Plus: symmetric bold disagreement (crop hi-conf violation vs
   main fully-high compliant -> review, matching the stated principle); crop FAILs act only with a
   high-confidence violation observation; the crop's image_quality_notes stay inside the warning
   dict (they described our own synthetic crop and triggered image-quality reframing of unrelated
   fields); abandoned crop threads now stop at the next pipeline stage via a threading.Event; OCR
   errors no longer abort the multi-image locate loop; eval/run_eval.py loads the Cloud Vision key
   (a flag-on eval without it silently scored the fallback); benchmark fixes (vacuous trust gate,
   unguarded futures, JPEG-as-PNG MIME). 173 unit tests pass.

   **Real-backs 5x verdict (the enable/don't-enable measurement)** —
   `scripts/benchmarks/real_backs_flag_benchmark.py` (output/real_backs_flag_benchmark_*): 13 real
   backs x 5 repeats, PAIRED (one extract_fields_with_warning_check call scores both arms — the
   debug full_warning IS the flag-off evidence — so the comparison has zero extraction noise).
   Flag OFF: PASS 13 / REVIEW 46 / FAIL 6 · flag ON: PASS 12 / REVIEW 47 / FAIL 6. **No review
   reduction on real labels** (+1 review->pass promotion on test_8, -2 pass->review demotions on
   test_11, a curved can the crop reads worse than the full image; net -1 pass). Safety held
   perfectly: 0 suspicious passes (full-FAIL->PASS structurally impossible), 0 added fails (the 6
   FAILs are identical in both arms: test_2 = the suspected REAL all-bold-body violation, body_bold
   True[high] 3/5; test_6 = pre-existing wording-read instability on a hard photo, untouched as
   text/caps), the concrete-doubt guard correctly ignored 6 medium-confidence crop reviews that
   would otherwise have demoted main passes, and FAIL-protection fired 4x. Crop used 39/65,
   discarded-slow 14/65 (~22% — real-back crops often lose the race), crop-call 500s 2/65 (clean
   fallback), disagreement 0/65. Latency p50 6.21 / p95 9.08 / max 9.34s; max wall-clock delta vs
   the main call 0.116s — never extended. **Conclusion: the flag STAYS OFF** — on real photographed
   labels the crop pass costs a second model call for ~zero review reduction; its measured value
   (all-bold-body detection, the disagreement safety net) is already mirrored by the fail-closed
   gate routing uncertainty to review. Revisit only if crop read confidence on real labels improves
   (the crop was rarely fully-high-confidence there, which is what review->pass requires).

review-burden conclusion is unchanged.

### Ensemble-size experiment — how many model reads per label? (tested the TA's multi-model idea)

Tests the TA's multi-model recommendation: do 2–5 parallel readers beat one? Four readers run
independently — **A** production (`extract_fields`, gpt-5.4-mini), **B** minimal-prompt (gpt-5.4-mini),
**C** warning-specialist (gpt-5.4-mini, warning-only), **D** higher-accuracy (`gpt-5.5`); **E** Gemini
skipped (no `GEMINI_API_KEY`). Ensemble sizes k=1..4 and four policies (single_best, majority_vote,
conservative_pass, reviewer_triage) are **simulated** from the collected reads, scored through the
unchanged `_check_warning`. `scripts/benchmarks/ensemble_size_experiment.py`, 41 products, 2 repeats on
the main set, 0 errors. Results separated: main label task (primary) vs bold_safety (secondary).
Artifacts: `artifacts/ensemble_size_experiment_results.{md,json}`.

| k | main false-pass known-bad | main false-fail (triage) | main review rate | par latency avg/max | cost× |
|---|---|---|---|---|---|
| 1 | 0/3 | **0/18** | 76% | 5.6 / 10.3 | 1.0 |
| 2 | 0/3 | 0/18 | 81% | 6.0 / 10.3 | 2.0 |
| 3 | 0/3 | 0/18 | 81% | 6.0 / 10.3 | 2.6 |
| 4 | 0/3 | 0/18 | 86% | **10.0 / 26.4** | **10.3** |

**The lever is the verifier policy, not the ensemble.** On the main task, false-passes are 0% at every
k; the false-fail fix comes from the **conservative / reviewer-triage Pass/Review/Fail policy at k=1**
(17%→0% just by switching policy), not from adding readers. More readers only raise review rate (62→86%)
and cost (up to 10×); the `gpt-5.5` reader (k=4) blows the latency budget (every product >5s, max 26s).
The **only** place k>1 helped was the secondary bold stress test: **k=2 (A+B) eliminated a single 1/15
not-bold false-pass** that one reader missed and halved verdict flips (6/21→2/21). k≥3 added nothing.

**Conclusion (TA recommendation not adopted):** the best production choice is **one reader + a
conservative Pass/Review/Fail verifier** — which is exactly what the production `header_body_gate`
already does (route uncertain bold → review). A **two-reader fallback (A + B, same cheap model)** is
reserved for optional bold-format hardening only; **three or more readers are unnecessary** (no
safety gain, multiplied cost/latency/review). **No production change** is warranted.

The `_PROMPT` `name_and_address` field description was widened (2026-06-09) to handle a name/address
printed as a SPLIT / STACKED block, and to nudge presence on a partial read. The prior wording
assumed a single "name and address line"; on `baseline_2` (Malt & Hop — back label stacked as
`BREWED & BOTTLED / By / MALT & HOP / BREWERY / … / HYATTSVILLE, MD`) the structured extractor
returned `present=false` (no address) on ~40% of runs, even though the raw-text OCR read the same
pixels fine — so the gap was structured-field ASSEMBLY, not resolution.

New wording (excerpt): *"Look across the ENTIRE label for the relationship phrase (e.g. 'BREWED &
BOTTLED BY', 'BOTTLED BY', 'PRODUCED AND BOTTLED BY', 'DISTILLED BY', 'IMPORTED BY'), the company
name, and the city/state. These pieces may be separated, stacked, curved, or printed in different
parts of the same label — combine them into ONE verbatim value in reading order … If you can read
the relationship phrase, the producer name, OR the city/state, set present=true … (use confidence
'low' if only partial)."* Stays blind to application values; the brand wording was **not** changed
(the dropped-"BREWERY" was a one-off, and the verifier's `token_set_ratio` already passes a dropped
suffix at 100, so a brand edit would be low-value and risks pulling the bottler trade name — e.g.
`Lighthouse Vintners` — into the brand).

**Before/after, baselines 5×** (`output/stability_{before,after}.json`):
- `name_and_address` present: **baseline_2 3/5 → 5/5** (no more `None`) and now at HIGH confidence
  (was medium/low); baseline_1 / baseline_3 unchanged at 5/5. Total **13/15 → 15/15**.
- No regression: brand sets unchanged (baseline_3 actually tightened to just `LIGHTHOUSE`, no
  `Lighthouse Vintners` bleed), class unchanged, warning bold unchanged (its own separate flakiness),
  extraction time **5.50s → 5.24s** avg.

**Real-label regression gate** — `test_labels/real_labels` 5× × 13 = **65 reads**
(`output/real_stability_after.json`):
- `name_and_address` present **60/65** (only `test_11`, a High Noon seltzer can, misses 0/5 — tiny
  curved can text, a read-difficulty miss, not over-assembly).
- **OVER-ASSEMBLY (URL / warning text / marketing copy pulled into the address): 0/65** by the
  deterministic detectors. Manual eyeball of all 13 address sets: clean — and the edit correctly
  ASSEMBLED complex multi-party statements a single-line prompt would fragment (`test_3` produced +
  bottled at two sites; `test_9` `PACKED FOR XXL BRANDS EUROPE … IMPORTED & DISTRIBUTED BY TRI-VIN
  IMPORTS …`; `test_12`/`test_13` Cervecería Modelo + Crown Imports). The long values are legitimate
  address content, not over-assembly.
- One minor blip: `test_10` (Bud Light Chelada) returned the Clamato trademark notice instead of the
  bottler on **1/5** runs (4/5 correct), and a later baseline 10× pass leaked a URL + net contents
  into one address (1/30) — both over-assembly side effects of the new "look across the ENTIRE label"
  wording. **Fixed by the exclusion-clause follow-up below.**
- Warning text/caps untouched: present 65/65, caps=True 65/65, text non-empty 65/65. Timing avg 6.71s.

**Eval after the change:** caught 9/11, 0 unexpected, **all three clean baselines overall=PASS** (the
cleanest completeness run; `clean_malt` now passes because its address reads). Unit tests 145 passed,
2 xfailed.

**Decision: KEPT the edit** — no serious real-label regression, eval sane, baselines improved. Only
`extraction.py` `_PROMPT` changed; bold / model choice / detail level / verifier logic untouched.

**Follow-up — exclusion clause (over-assembly fix).** The "look across the ENTIRE label" wording
occasionally over-reached (the `test_10` trademark, 1/65; and a baseline 10× pass that leaked a URL +
net contents into one address, 1/30). One narrow sentence was added to the `name_and_address` prompt:
*"Do not include URLs, website text, net contents, alcohol content, government warning text, slogans,
marketing copy, trademark notices, barcode text, or unrelated statements in name_and_address."*
Re-running the 10× baseline measurement: **leak flags 1/30 → 0/30**, baseline_2 `name_and_address`
stayed **present 10/10** (all three 10/10), brand/class **30/30 verifier-PASS**, warning wording
**exact 10/10** + caps **10/10**, bold flakiness unchanged, timing similar (avg ~5.5s, median 5.25s,
0 retries). Only the `name_and_address` prompt sentence changed; verifier / bold / model / detail
untouched. Unit tests 145 passed, 2 xfailed.
